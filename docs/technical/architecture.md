# 아키텍처

## 목표

Battle Hamsters는 다음 네 개의 영역이 명확히 분리된 구조를 목표로 한다.

- `apps/portal`: 포털, 로비, 매칭, 방 진입 UI
- `apps/game`: Phaser 기반 실시간 게임 클라이언트
- `packages/shared`: 타입, 이벤트 스키마, 공용 상수
- `server`: 서버 권위(authoritative) 실시간 게임 서버 및 API

## 시스템 경계

### Portal
책임:
- 닉네임 입력
- 방 생성/목록/입장 UI
- 매칭 상태 표시
- 게임 서버 또는 게임 클라이언트 진입에 필요한 세션 정보 전달

비책임:
- 실시간 전투 판정
- 플레이어 위치 계산
- 무기 판정

### Game Client
책임:
- 입력 수집
- 렌더링
- 카메라/HUD
- 서버 상태 보간
- 로컬 체감 개선(예측은 후순위)

비책임:
- 최종 이동 판정
- 최종 공격 판정
- 승패 확정

### Shared Package
책임:
- WebSocket 메시지 타입
- 맵/무기/플레이어 데이터 타입
- 공통 상수와 enum

### Server
책임:
- 룸/세션 관리
- 입력 처리
- 이동 판정
- 공격 판정
- HP/생명/낙사 처리
- 결과 확정

## 데이터 흐름

1. Portal에서 플레이어가 닉네임과 방 정보를 설정한다.
2. Portal은 방 정보 또는 플레이 세션 토큰을 만든다.
3. Game Client는 해당 세션 정보로 서버 WebSocket에 연결한다.
4. Server는 룸 참가를 승인하고 스냅샷 전송을 시작한다.
5. Game Client는 입력을 전송하고, Server는 authoritative 상태를 브로드캐스트한다.

## 네트워크 원칙

- 서버 권위 모델 사용
- 클라이언트는 입력만 제안하고 상태는 서버가 확정
- 초기 단계에서는 단순성과 일관성을 우선
- 고급 예측/보간은 Phase 2 이후 확장

## MVP 수직 슬라이스

첫 번째 구현 목표는 아래 한 줄이다.

> Portal에서 방에 들어가고, Game Client가 서버에 연결해서, 더미 또는 실제 플레이어 상태를 수신하며 이동 입력을 주고받을 수 있다.

### 포함 범위
- Portal 단일 로비 화면
- 방 코드 또는 단일 테스트 룸 입장
- Server WebSocket 연결
- 플레이어 join/leave
- 입력 전송
- 서버 스냅샷 브로드캐스트
- Game Client에서 플레이어 위치 렌더링

### 제외 범위
- DB 기반 계정
- 랭크 매칭
- 전투 판정
- 재접속 복구
- 관전자 모드

## 단계별 구현 순서

### Milestone 0: 연결 확인
- Game Client가 서버에 연결된다.
- 서버가 `welcome`과 `room_snapshot`을 보낸다.
- 클라이언트가 자기 자신과 더미 플레이어를 렌더링한다.

### Milestone 1: 이동 동기화
- 클라이언트가 입력을 보낸다.
- 서버가 authoritative 위치를 계산한다.
- 모든 클라이언트가 동일한 위치를 본다.

### Milestone 2: 기본 전투
- 기본 무기(Paws, Acorn Blaster) 추가
- 사격 이벤트 처리
- HP 감소 및 리스폰

### Milestone 3: 룸/매치 흐름
- 방 생성/입장/퇴장
- 매치 시작/종료
- 결과 화면

## 리스크

1. **Portal과 Game의 경계 불분명**
   - 해결: Portal은 매치 전 UI, Game은 실시간 전투에 집중
2. **서버/클라이언트 메시지 형식 드리프트**
   - 해결: `packages/shared`에 타입 집중
3. **너무 빠른 기능 확장**
   - 해결: 무기/상태이상/매칭은 단계적으로 추가
