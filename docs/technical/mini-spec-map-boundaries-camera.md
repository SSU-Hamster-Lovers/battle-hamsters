# 미니 스펙 — 맵 경계 정책과 카메라 규칙

## 작업명

맵 경계 정책과 카메라 규칙 정의

## 목적

맵 컨셉에 따라 **폐쇄형 맵 / 개방형 맵**을 구분하고,
향후 카메라 이동, 플레이 가능 범위, 화면 밖 이동, out-of-bounds 사망 규칙을 일관되게 설계하기 위한 기준을 미리 정한다.

## 왜 필요한가

현재 테스트 맵은 좌우가 닫힌 단순 구조를 사용하고 있지만,
실제 게임에서는 다음과 같은 요구가 생길 수 있다.

- 좌우 벽이 있는 폐쇄형 맵
- 화면 밖으로도 이동 가능한 개방형 맵
- 플레이어 위치에 따라 카메라가 이동하는 맵
- 큰 맵 / 긴 맵 / 동적 카메라가 필요한 맵
- 논리적 울타리와 시각적 울타리가 다른 맵

이 규칙을 미리 정하지 않으면,
나중에 맵 구현과 카메라 구현이 서로 다른 가정을 가지게 될 수 있다.

## 핵심 결론

맵은 좌우 경계 정책에 따라 최소 두 종류를 지원할 수 있어야 한다.

### 1. 폐쇄형 맵 (`closed`)

- 좌우 끝에 solid wall 존재
- 플레이어는 화면 밖으로 나갈 수 없음
- 근거리 난전, 압박형 구조에 적합

### 2. 개방형 맵 (`open`)

- 좌우 벽이 없음
- 플레이어는 화면 렌더링 범위 밖으로도 일정 범위 이동 가능
- 다만 너무 멀리 벗어나면 out-of-bounds 판정으로 사망 또는 리스폰 처리 가능
- 허공 발판, 측면 우회 루트, 넓은 전장 구조에 적합

## 경계 관련 개념 분리

향후 맵 설계에서는 아래 범위를 구분하는 것이 좋다.

### 화면 렌더링 범위 (`camera viewport`)

- 현재 카메라가 실제로 보여주는 범위
- 플레이 가능 범위와 같을 수도 있고 다를 수도 있음

### 시각적 울타리 (`visual bounds`)

- 카메라가 **절대 넘어가서 보여주면 안 되는 최대 시각 범위**
- 카메라 clamp의 기준
- 실제 맵 데이터는 이 바깥에도 존재할 수 있지만, 카메라는 여기까지만 보여준다

### 플레이 가능 범위 (`gameplay bounds`)

- 플레이어가 실제로 이동 가능한 범위
- 개방형 맵에서는 화면 밖까지 포함될 수 있음

### 사망 범위 (`death bounds` / `out-of-bounds`)

- 너무 멀리 벗어나면 사망 처리되는 범위
- 아래쪽 낙사(`fall zone`)와는 별도로 좌우 바깥 범위도 정의 가능

### 해석 원칙

- pit 아래 낙사 유도 영역, 버그 방지용 fence 같은 요소는 **논리적 울타리**로 본다.
- 카메라는 이런 논리적 울타리 영역을 보여주지 않는다.
- 즉, **카메라는 visual bounds까지만**, 시스템 판정은 gameplay/death bounds까지 볼 수 있다.

## 카메라 정책 초안

현재는 정적인 테스트 카메라로 충분하지만, 향후 아래 정책을 고려한다.

### 1. 정적 카메라

- 테스트 맵과 소형 아레나에 적합
- 현재 구현과 가장 단순하게 맞음

### 2. 맵 기반 카메라 이동

- 카메라가 플레이어 또는 전투 중심을 따라 움직임
- 큰 맵, 측면 진행형 구조, 개방형 맵에 적합
- 카메라는 `visualBounds`를 넘어가지 않는다
- pit 아래 낙사용 영역, 버그 방지용 논리 fence 같은 영역은 보여주지 않는다

### 3. 동적 카메라 범위 조절

- 플레이어 간 거리나 전장 상황에 따라 화면이 넓어지거나 이동함
- 다인전에서는 복잡도가 커지므로 후순위

## 카메라 이동 감각 원칙

- 카메라 이동은 선형 고정 속도보다 **지연 추적 + 가속/감속이 있는 자연스러운 이동**을 선호한다.
- 기본 후보는 `smooth damp`, `spring`, `critical damping` 같은 감쇠 기반 추적이다.
- **Bezier 곡선은 기본 follow 카메라 수단으로 고정하지 않는다.**
  - 실시간 추적보다는 스크립트 연출/특수 경로 이동 쪽에 더 적합하다.

## 다인전 카메라 주의점

`모든 플레이어를 한 화면에 보여줄 것인가?` 는 별도 정책이 필요하다.

현재는 다음 중 하나를 나중에 선택할 수 있다.

1. **고정 카메라 / 로컬 시점 기반**
2. **모든 플레이어를 화면 안에 넣는 줌 조절 카메라**
3. **맵별 카메라 정책 선택**

초기에는 복잡도를 줄이기 위해 **고정 또는 단순 추적 카메라**를 우선 권장한다.

## 데이터 관점 초안

장기적으로는 맵 데이터에 아래 정책이 들어갈 수 있다.

```json
{
  "boundaryPolicy": "closed",
  "cameraPolicy": "static",
  "visualBounds": {
    "left": 0,
    "right": 1280,
    "top": 0,
    "bottom": 720
  },
  "gameplayBounds": {
    "left": 0,
    "right": 1280,
    "top": 0,
    "bottom": 720
  },
  "deathBounds": {
    "left": -200,
    "right": 1480,
    "top": -9999,
    "bottom": 900
  }
}
```

### 예시 필드 의미

- `boundaryPolicy`: `closed | open`
- `cameraPolicy`: `static | follow | dynamic`
- `visualBounds`: 카메라가 보여줄 수 있는 최대 시각 범위
- `gameplayBounds`: 실제 이동 가능 범위
- `deathBounds`: out-of-bounds 사망 범위

## 현재 브랜치와의 관계

이 미니 스펙은 **현재 브랜치의 직접 구현 범위가 아니다.**
shared-map-json 작업은 맵 데이터 원본 통합이 목적이고,
실제 카메라 follow 구현은 후속 작업이다.

즉 이 문서는 **지금 구현할 것**이 아니라,
**다음 맵/카메라 확장 시의 기준을 미리 고정하기 위한 문서**다.

## 현재 권장 방침

- 지금 브랜치에서는 공통 맵 JSON 기준선을 먼저 만든다.
- 맵 경계 정책과 카메라 이동 런타임 구현은 별도 브랜치에서 진행한다.
- 다만 앞으로의 맵 JSON 설계 시 `boundaryPolicy`, `cameraPolicy`, `visualBounds`, `gameplayBounds`, `deathBounds`를 고려한다.
