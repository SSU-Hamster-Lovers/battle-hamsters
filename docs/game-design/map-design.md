# 맵 설계

## 데이터 포맷

**JSON** 사용.

| 장점 | 설명 |
|------|------|
| 범용성 | Phaser (JS) + Rust (serde_json) 모두 호환 |
| 도구 | Tiled 에디터 등이 JSON 지원 |
| 버전관리 | diff로 변경 사항 추적 용이 |

## 기본 구조

```json
{
  "id": "arena_01",
  "name": "기본 아레나",
  "width": 1280,
  "height": 720,
  "spawnPoints": [
    { "x": 100, "y": 360 },
    { "x": 1180, "y": 360 }
  ],
  "terrain": [...],
  "collision": [...],
  "hazards": [...],
  "weaponSpawns": [...],
  "itemSpawns": [...]
}
```

## 구성 요소

| 요소 | 설명 |
|------|------|
| `terrain` | 지형 (바닥, 벽 등) |
| `collision` | 충돌 영역 |
| `spawnPoints` | 스폰 위치 |
| `weaponSpawns` | 무기 스폰 위치 |
| `itemSpawns` | 아이템 스폰 위치 |
| `hazards` | 위험 구역 (낙사 등) |
| `boundaryPolicy` | 맵 좌우 경계 정책 (`closed | open`) |
| `cameraPolicy` | 카메라 정책 (`static | follow | dynamic`) |
| `gameplayBounds` | 실제 이동 가능 범위 |
| `deathBounds` | out-of-bounds 사망 범위 |
| `decorations` | 장식 (시각적 요소) |

## 설계 원칙

- 공정성: 스폰 위치 대칭
- 다양성: 다양한 전략 가능
- 스폰킬 방지: 안전한 스폰 위치
- 기동성: 빠른 이동과 점프 루트가 살아 있어야 함
- 순환성: 무기 먹기/버리기/재확보 루프가 자연스럽게 일어나야 함
- 경쟁성: 무기 스폰과 아이템 스폰이 서로 다른 이동 동선을 만들어야 함

## 무기 스폰 원칙

- 맵 스폰 무기는 플레이어 이동 루프를 유도해야 한다.
- 너무 안전한 위치에만 몰아두지 않는다.
- 스폰된 무기와 버려진 무기는 일정 시간이 지나면 사라져야 한다.
- 초기 기준 despawn 시간은 약 **10초**를 목표로 한다.

### 무기 스폰 연출 예시
- 공중 스폰 → 중력 낙하
- 필드 스폰 → 페이드인
- 오브젝트 트리거 스폰 → 코코넛, 상자 등에서 등장

### 무기 디스폰 연출
- 작아지면서 뿅하고 사라지는 연출을 기본 기준으로 둔다.

## 아이템 스폰 원칙

- 아이템 스폰은 무기 스폰과 다른 경쟁 포인트를 만들어야 한다.
- 이동/생존/점프 확장 아이템은 루트 선택을 바꾸는 위치에 배치한다.
- 점프 아이템은 고지대 접근, 우회 루트, 탈출 루트와 연동될 때 가치가 높다.


## 플랫폼 이동을 위한 맵 규칙

- 일반 바닥/벽 충돌과 원웨이 플랫폼을 구분해야 한다.
- 일반 바닥은 `floor surface`, 좌우 이동 차단은 `solid wall` primitive로 구분한다.
- 원웨이 플랫폼은 위에서 착지 가능하지만 아래에서는 통과 가능해야 한다.
- `아래 + 점프` 입력 시 잠시 플랫폼 충돌을 무시하는 규칙을 지원해야 한다.
- 바닥이 없는 구간 아래에는 `fall zone`을 둘 수 있어야 한다.
- 낙사와 별개로 가시/트랩 같은 `instant kill hazard`도 별도 배치 가능해야 한다.
- 리스폰 위치는 플레이어가 상공에서 다시 떨어질 수 있는 구조가 좋다.

## 맵 경계 정책

### `closed`
- 좌우 끝에 wall이 있거나, gameplay bounds를 벗어나지 못하게 설계된 맵
- 카메라는 고정 아레나형 구조와 궁합이 좋다.

### `open`
- 좌우 화면 밖으로도 이동 가능한 맵
- 대신 `deathBounds`를 별도로 둬서 너무 멀리 벗어나면 사망 처리할 수 있다.

## 카메라 정책

### `static`
- 테스트 맵, 소형 아레나, 단순 난전에 적합
- 현재 구현이 가장 쉽게 맞출 수 있는 기본값

### `follow`
- 플레이어 또는 전투 중심을 따라가는 카메라
- 개방형 맵과 큰 맵의 기본 후보

### `dynamic`
- 플레이어 간 거리나 전장 상황에 따라 카메라 범위가 바뀌는 정책
- 다인전에서는 복잡도가 높아 후순위

## 맵 제작 주의점

- pit이나 측면 낙하 루트는 **복귀 가능성 자체를 시스템으로 막기보다 맵 설계 의도**로 다루는 쪽을 우선한다.
- 즉, 낙사 구간 깊이, 벽 연장 길이, 복귀용 플랫폼/우회 루트 배치는 맵별로 결정한다.
- 추후 open 맵을 만들 때는 `camera viewport`, `gameplayBounds`, `deathBounds`를 분리해서 검토한다.
