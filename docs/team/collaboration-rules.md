# 협업 규칙

## 핵심 결론

현재 팀 구성에서는 **팀장이 v0.1 수직 슬라이스를 먼저 만드는 전략**이 효율적이다.
다만 이것이 "팀장이 모든 걸 혼자 만든다"는 뜻은 아니다.

권장 방식은 아래와 같다.

> 팀장이 핵심 구조와 최소 플레이 가능 버전을 먼저 고정하고,
> 다른 개발자와 비개발자는 병렬 가능한 자산, 문서, 테스트, 저위험 구현을 지원한다.

## 왜 이 방식이 맞는가

### 장점

1. 초기 방향이 흔들리지 않는다.
2. Portal / Game / Server 경계가 빠르게 고정된다.
3. 코드 스타일과 메시지 계약이 한 번에 정리된다.
4. 작은 팀에서 커뮤니케이션 비용을 줄일 수 있다.

### 주의점

1. 팀장이 병목이 되기 쉽다.
2. 다른 팀원이 프로젝트 맥락을 못 배우고 대기 상태가 될 수 있다.
3. "나중에 합류"한 팀원의 온보딩 비용이 커질 수 있다.

따라서 **혼자 다 만들기**보다 **핵심만 직접 만들고, 나머지는 일찍 분배하는 구조**가 좋다.

## 단계별 운영 전략

## Stage A — 설계 및 계약 고정

이 단계는 팀장이 주도한다.

### 팀장 책임
- MVP 범위 확정
- 문서 기준선 작성
- 폴더 구조/브랜치 전략 정리
- 기술 계약 확정
  - 데이터 포맷
  - WebSocket 메시지 타입
  - 최소 게임 루프

### 병렬 위임 가능 작업

#### 협업 개발자
- 문서 리뷰
- 실행 가이드 검증
- 타입 초안 정리 보조
- 테스트/린트/CI 보조

#### 비개발자
- 무기 아이디어 시트 정리
- 맵 레퍼런스 수집
- UI 문구/로비 문구 정리
- 플레이 테스트 체크리스트 초안 작성

## Stage B — v0.1 수직 슬라이스 구현

목표:
- 방 입장
- 서버 연결
- 위치 동기화
- 기본 플레이어 렌더링

### 팀장 책임
- shared 계약 정의
- server WebSocket 기본 구조
- game 연결 핵심 루프
- 전체 통합

### 협업 개발자 책임
- 타입 보강
- 디버그 UI
- 로그 정리
- 버그 수정
- 반복 테스트

### 비개발자 책임
- 임시 에셋 정리
- 기본 HUD 문구
- 테스트 케이스 작성
- 버그 제보 양식 관리

## Stage C — 확장 및 생산 분산

v0.1 이후에는 작업을 더 분산한다.

- 팀장: 구조 변경이 필요한 작업, 중요 우선순위, 리뷰
- 협업 개발자: 기능 추가, 버그 수정, 툴링 정리
- 비개발자: 맵, 무기 데이터, 아트, 사운드, QA 루프 강화

## 브랜치 운영 규칙

### 기본 원칙
- `main`: 배포 가능 상태
- `develop`: 통합 브랜치
- `docs/*`: 문서 작업
- `feature/*`: 기능 개발
- `fix/*`: 버그 수정

### 권장 규칙
1. 문서 브랜치에는 구현 코드를 섞지 않는다.
2. 하나의 브랜치는 하나의 목적만 가진다.
3. 팀장이 고정한 계약을 바꾸는 PR은 반드시 사전 합의한다.
4. 큰 기능은 바로 개발하지 말고 먼저 문서 또는 이슈로 쪼갠다.

## 작업 분배 규칙

### 팀장이 직접 잡아야 하는 작업
- 시스템 경계 결정
- 네트워크 프로토콜 변경
- shared 타입의 breaking change
- 첫 구현 패턴 정립
- 최종 통합

### 협업 개발자에게 빨리 넘길 수 있는 작업
- UI 마크업
- 로깅/디버그 패널
- 단위 테스트
- 문서 반영형 타입/상수 추가
- 버그 재현 및 수정
- 반복적 리팩터링

### 비개발자에게 바로 넘길 수 있는 작업
- 무기 설명표
- 맵 컨셉 시트
- HUD 문구 / 튜토리얼 문구
- 에셋 목록표
- 테스트 체크리스트
- 플레이 피드백 정리

## 회의 / 커뮤니케이션 규칙

### 짧은 주기 권장
- 매일 또는 격일 10~15분 스탠드업
- 주 1회 범위/우선순위 재정렬

### 스탠드업 질문
1. 어제 한 일
2. 오늘 할 일
3. 막힌 점

### 의사결정 규칙
- 규칙, 프로토콜, 구조 변경은 팀장이 최종 결정
- 콘텐츠, 아트, UX 톤은 담당자가 초안 제시
- 합의가 길어지면 MVP 기준으로 더 단순한 쪽을 선택

## 리뷰 규칙

1. PR은 작게 유지한다.
2. 작성자와 리뷰어 역할을 분리한다.
3. 문서 변경도 PR로 남긴다.
4. "좋아 보인다"보다 "무엇을 검증했는지"를 남긴다.
5. 버그 수정 PR에는 재현 조건과 해결 조건을 적는다.

## 안티패턴

아래 방식은 피한다.

- 모든 일을 팀장이 직접 오래 붙잡는 것
- 비개발자를 후기 단계에만 투입하는 것
- 문서 없이 바로 병렬 구현하는 것
- 한 브랜치에 문서/기능/리팩터링을 모두 섞는 것
- 아무도 오너가 아닌 작업을 만드는 것

## 현재 팀에 대한 실전 권고

당신의 생각처럼 **팀장이 1차 버전까지 중심을 잡는 방향은 맞다.**
하지만 더 정확히는 아래가 좋다.

- **팀장이 v0.1의 핵심 골격을 만든다**
- **다른 개발자는 병렬 가능한 저위험 구현과 디버깅을 맡는다**
- **비개발자 3명은 에셋, 콘텐츠, 테스트 자산을 즉시 생산한다**

즉,

> "팀장이 다 만든 뒤 나머지가 붙는다" 보다는
> "팀장이 중심 골격을 만들고, 나머지는 초반부터 병렬 자산을 쌓는다"가 더 효율적이다.

## 스펙 점검 규칙

- 작업 시작 전 현재 스펙과 현재 구현 상태 문서를 확인한다.
- 구현 전 미니 스펙을 짧게 작성한다.
- PR 머지 전 스펙 문서와 구현 상태 문서를 다시 점검한다.
- 결과물은 기본적으로 한국어로 정리한다.
