# 스펙 주도 개발 워크플로

## 목적

이 문서는 Battle Hamsters에서 구현 작업을 할 때, 어떤 AI 도구를 쓰더라도 공통으로 따를 수 있는 **스펙 주도 개발 절차**를 정의한다.

## 핵심 원칙

1. 구현 전에 스펙을 확인한다.
2. 구현 중 스펙과 코드가 어긋나면 즉시 문서나 코드를 맞춘다.
3. 임시 구현도 문서에 남긴다.
4. PR 머지 전에는 반드시 스펙 상태를 다시 점검한다.

## 작업 시작 전 체크리스트

- [ ] 관련 게임 스펙 문서를 읽었는가?
- [ ] 관련 기술 스펙 문서를 읽었는가?
- [ ] `current-implementation.md`를 확인했는가?
- [ ] 이번 작업의 범위와 비목표를 정했는가?
- [ ] 검증 방법을 정했는가?

## 미니 스펙 템플릿

```text
작업명:
목표:
이번 범위:
건드리는 스펙/문서:
비목표:
검증 방법:
```

## 작업 중 규칙

- 스펙 변경이 생기면 구현보다 먼저 영향 범위를 적는다.
- 서버/클라이언트/shared 중 둘 이상에 걸치는 변경은 `docs/technical/data-formats.md`와 `docs/technical/sync-protocol.md`를 우선 확인한다.
- 게임 규칙이 바뀌면 `docs/game-design/*`도 같이 갱신한다.

## PR 머지 전 체크리스트

- [ ] 코드와 문서가 같은 내용을 말하는가?
- [ ] 스펙 영향이 있었는데 문서가 빠진 곳은 없는가?
- [ ] 현재 구현 상태 문서가 최신인가?
- [ ] 테스트/타입체크/수동 검증 결과를 남겼는가?
- [ ] 임시 구현과 다음 단계가 명시되어 있는가?

## 문서 갱신 기준

### 반드시 갱신
- 프로토콜
- 데이터 필드
- 무기/아이템 규칙
- 이동/점프 규칙
- 실행 방법
- 협업 규칙

### 선택적 갱신
- 순수 리팩터링
- 내부 구조 정리
- 테스트 추가만 있는 경우
