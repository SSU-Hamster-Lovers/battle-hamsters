## 문제 요약

현재 플레이어의 플랫폼 하강은 `dropThroughUntil` 시간 창 동안 **모든** `one_way_platform` 충돌을 무시하는 방식이다.

이 방식은 아래 두 조건이 겹치면 불안정해진다.

- 플랫폼 사이 간격이 좁다.
- 서버가 20 TPS라 한 틱 이동량과 무시 시간 창이 상대적으로 크다.

그 결과, 플레이어는 의도한 첫 번째 플랫폼만 내려가는 대신 그 아래의 두 번째 플랫폼까지 한 번에 통과할 수 있다.

## 현재 구조의 핵심 문제

`server/src/room_runtime.rs`

- 하강 시작 시: `drop_through_until = now_ms + DROP_THROUGH_MS`
- 착지 판정 시: `drop_active`가 true면 모든 one-way platform 착지를 건너뜀

즉 현재 상태는 "내가 방금 내려온 그 플랫폼만 무시"가 아니라,
"일정 시간 동안 세상 모든 원웨이 플랫폼을 무시"다.

이 구조에서는 플랫폼 간격이 좁을수록 버그가 더 자주 보이지만, 근본 원인은 간격 자체보다도 **전역 시간 무시 정책**이다.
20 TPS는 이 현상을 더 쉽게 드러나게 만들 뿐, 유일한 원인은 아니다.

## 원인 분해

### 1. 전역 무시 범위가 너무 넓다

- 원하는 동작: 현재 서 있는 플랫폼 1개만 잠시 무시
- 실제 동작: 일정 시간 동안 아래의 다른 플랫폼도 전부 무시

### 2. 시간 기반 종료는 지형 밀도에 둔감하다

- 플랫폼 사이 간격이 넓으면 운 좋게 두 번째 플랫폼 위에서 시간이 끝날 수 있다.
- 플랫폼 사이 간격이 좁으면 같은 시간 창 안에 두 번째 플랫폼 top surface까지 도달한다.

### 3. 20 TPS는 완화 대상이지 본질 해결책이 아니다

- TPS를 높이면 재현 빈도는 줄 수 있다.
- 하지만 "전역 무시" 구조가 그대로면 가까운 플랫폼 조합에서 같은 종류의 버그는 계속 남는다.

## 해결 옵션

### 옵션 A — `DROP_THROUGH_MS`만 줄인다

장점:
- 구현이 가장 쉽다.

단점:
- 플랫폼 간격/하강 속도/넉백 상태에 따라 다시 깨진다.
- 맵 구조에 따라 맞는 시간이 계속 달라진다.

결론:
- 임시 완화는 가능하지만 구조적 해결책은 아니다.

### 옵션 B — 무시 대상을 "현재 플랫폼 1개"로 한정한다

제안 상태:

```text
dropThroughPlatformId: string | null
dropThroughUntil: number | null
```

동작:
- 하강 시작 시, 현재 서 있던 플랫폼의 ID를 기억한다.
- 착지 판정에서는 그 플랫폼만 잠시 무시한다.
- 다른 플랫폼은 같은 틱부터 정상 착지 후보로 유지한다.

장점:
- 바로 아래 두 번째 플랫폼은 정상적으로 다시 잡힌다.
- 플랫폼 간격이 좁아도 의도한 문법과 맞는다.

단점:
- 현재 런타임 맵 primitive에 stable id 접근이 필요하다.
- 상태 필드가 하나 더 늘어난다.

결론:
- v1 구조 해법으로 가장 적절하다.

### 옵션 C — 플랫폼 ID + "원본 플랫폼 아래로 충분히 내려갔는지" 거리 조건을 같이 쓴다

추가 규칙:
- `playerBottom > sourcePlatformTopY + clearMargin` 이 되면 source 플랫폼 무시를 해제
- 또는 source 플랫폼의 x 범위를 벗어나면 해제

장점:
- 하강 중 다시 원래 플랫폼에 붙는 떨림을 줄인다.
- 짧은 시간창보다 지형 기반이라 설명 가능성이 높다.

단점:
- 구현이 옵션 B보다 조금 더 복잡하다.

결론:
- 실제 구현은 B 단독보다 B+C 조합이 더 안전하다.

## 권장 방향

### 권장안: B + C 조합

1. 하강 시작 시 현재 밟고 있는 one-way platform의 `id`를 저장한다.
2. 착지 판정에서 저장된 source platform만 무시한다.
3. 플레이어 bottom이 source platform 아래로 일정 거리 내려가면 source ignore를 해제한다.
4. source가 아닌 다른 one-way platform은 즉시 다시 착지 후보로 본다.

이렇게 하면:
- 첫 번째 플랫폼은 의도대로 통과
- 바로 아래 두 번째 플랫폼은 정상 착지
- TPS나 플랫폼 간격에 덜 민감

## 구현 시 필요한 데이터 변경

### 맵 런타임

- `OneWayPlatformSegment`에 `id` 추가

### 플레이어 상태

현재:

```text
dropThroughUntil: Option<u64>
```

후보:

```text
dropThroughUntil: Option<u64>
dropThroughPlatformId: Option<String>
```

또는 내부 런타임 전용 필드로 관리하고 스냅샷에는 굳이 노출하지 않는 방법도 가능하다.

## 구현 시 주의점

- 하강 시작 직후 플레이어를 강제로 조금 아래로 내리는 보정(`position.y += 2.0`)은 유지 가능하지만, source platform ignore와 함께 검토해야 한다.
- 넉백/공중 급강하 중에도 같은 규칙이 예측 가능해야 한다.
- 플랫폼 여러 개가 x축으로 겹쳐 있는 경우, 현재 접촉 중인 플랫폼 식별 규칙을 명확히 해야 한다.

## 결론

이 문제는 "플랫폼 간격이 좁아서만"도 아니고, "20 TPS라서만"도 아니다.
핵심은 현재 하강 로직이 **전역 시간 무시** 기반이라서, 가까운 원웨이 플랫폼 체인에서 두 번째 플랫폼까지 같이 잃어버리는 구조라는 점이다.

따라서 다음 구현은 `DROP_THROUGH_MS` 튜닝보다,
"현재 밟고 있던 source platform만 무시"하는 식의 **플랫폼 단위 drop-through 상태**로 가는 것이 맞다.
