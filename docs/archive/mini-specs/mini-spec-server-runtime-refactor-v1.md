# 미니 스펙 — 서버 런타임 구조 분리 리팩토링 1차

## 작업명

서버 런타임 구조 분리 리팩토링 1차

## 목표

`server/src/main.rs`에 과도하게 모여 있는 런타임 데이터 로딩, 룸 로직, 전투/아이템 로직을
작은 책임 단위로 나누어 이후 기능 작업이 더 안전하게 들어갈 수 있게 만든다.

## 이번 범위

- 동작 보존을 전제로 파일 분리 리팩토링 수행
- 최소 1차로 아래 우선 영역을 분리
  - 런타임 맵/무기/아이템 데이터 로딩
  - 이후 분리 후보: 룸 상태 / 전투 / 아이템 / WS 세션
- 기존 서버 테스트를 계속 통과시키며 진행

## 건드리는 스펙/문서

- `docs/technical/current-implementation.md`
- 필요 시 `docs/technical/architecture.md`

## 비목표

- 서버 아키텍처 전면 재설계
- trait 중심 추상화 대량 도입
- ECS 전환
- 네트워크 프로토콜 변경
- 룸/전투 규칙 변경

## cleanup plan

### 1. 행동 보존 기준 고정

- 기존 서버 테스트를 회귀 보호막으로 사용한다.
- 리팩토링 중 새 기능을 섞지 않는다.

### 2. 데이터/설정 영역부터 분리

- `training-arena.json`
- 무기 JSON
- 아이템 JSON
  을 읽는 런타임 데이터 로딩 코드를 먼저 분리한다.

### 3. 상태 변화 로직은 다음 단계에서 작은 덩어리로 분리

- room state
- weapon runtime
- item runtime
- ws session
  순으로 작은 책임 단위 분리를 검토한다.

### 4. 큰 리네임보다 작은 파일 추출을 우선

- 한 번에 전부 재배치하지 않는다.
- 컴파일/테스트가 자주 통과하는 작은 단계로 나눈다.

## 핵심 결정

### 1. 리팩토링은 다음 기능 작업을 위한 안전장치다

- room gameplay config
- spawn behavior
- random spawn candidates
  같은 후속 작업이 `main.rs` 단일 파일에 계속 쌓이지 않게 하는 것이 목적이다.

### 2. 테스트가 있는 상태에서만 구조를 분리한다

- 기능 보호 없이 대규모 이동은 하지 않는다.
- 현재 서버 테스트가 최소 보호막 역할을 한다.

### 3. 1차는 “분리”가 목표지 “완성된 구조”가 목표는 아니다

- 1차에서 모든 설계를 끝내지 않는다.
- 가장 큰 병목을 만드는 파일 응집도만 먼저 줄인다.

## 완료 조건

- `main.rs`에서 적어도 한 덩어리 이상의 책임이 별도 파일로 빠진다.
- 서버 테스트가 계속 통과한다.
- 이후 room config / spawn behavior 작업을 더 작은 진입점에서 시작할 수 있다.
