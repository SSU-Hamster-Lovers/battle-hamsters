# Battle Hamsters 에이전트 공통 작업 규칙

이 문서는 특정 도구 하나에 묶이지 않는 **공통 작업 규칙**이다.
Codex, Claude Code CLI, Kilo 코드 등 어떤 AI 에이전트 도구를 사용하더라도 아래 규칙을 우선적으로 따른다.

## 1. 언어 규칙

- 내부 추론 언어는 도구가 자유롭게 선택할 수 있다.
- 하지만 **사용자에게 보이는 결과물은 기본적으로 한국어**로 작성한다.
- 예외:
  - 코드 문법
  - API/라이브러리 식별자
  - 타입 이름
  - 파일 경로
  - 외부 서비스 고유 명칭
- PR 제목/본문, 이슈 설명, 작업 요약, 문서, 주석성 설명은 특별한 사유가 없으면 한국어를 우선한다.

## 2. 작업 시작 전 규칙

실제 구현이나 수정 전에 반드시 아래를 점검한다.

1. 현재 스펙 문서 확인
   - `docs/game-design/core-rules.md`
   - `docs/game-design/item-design.md`
   - `docs/game-design/map-design.md`
   - `docs/game-design/weapon-design.md`
   - `docs/technical/architecture.md`
   - `docs/technical/data-formats.md`
   - `docs/technical/sync-protocol.md`
2. 현재 구현 상태 확인
   - `docs/technical/current-implementation.md`
3. 작업 범위에 영향을 주는 스펙 문서가 무엇인지 먼저 식별
4. 구현 전 **미니 스펙**을 짧게 작성

## 3. 미니 스펙 규칙

모든 구현 작업은 시작 전에 아래 형식으로 간단히 정리한다.

```text
작업명:
목표:
이번 범위:
건드리는 스펙/문서:
비목표:
검증 방법:
```

- 길게 쓰기보다 **짧고 정확하게** 적는다.
- 구현만 하는 것 같아도, 스펙 영향이 있으면 반드시 문서 영향 범위를 적는다.

## 4. 스펙 주도 개발 규칙

- 구현보다 스펙이 먼저다.
- shared 계약, 서버 메시지, 게임 규칙, 아이템/무기 동작이 바뀌면 **문서를 같이 갱신**한다.
- “일단 구현하고 나중에 문서화”를 기본값으로 두지 않는다.
- 구현이 임시 동작이라면 문서에 **임시 구현**임을 명시한다.

## 5. PR 머지 전 점검 규칙

PR을 머지하기 전 반드시 아래를 확인한다.

1. 현재 코드가 어떤 스펙을 구현하는지 설명 가능해야 한다.
2. 스펙이 바뀌었다면 관련 문서가 함께 갱신되어 있어야 한다.
3. `docs/technical/current-implementation.md`가 현재 상태를 반영해야 한다.
4. 검증 결과(테스트, 타입체크, 스모크 테스트)를 남겨야 한다.
5. 임시 구현이면 다음 단계와 한계를 PR에 명시해야 한다.

## 6. 문서 갱신이 필요한 변경

아래 변경은 문서 갱신을 동반해야 한다.

- WebSocket 메시지 구조 변경
- shared 타입 변경
- 게임 규칙 변경
- 무기/아이템 동작 변경
- 이동/점프/전투 로직 변경
- 로컬 실행 방식 변경
- 협업 규칙 변경

## 7. 추천 문서 흐름

- 제품/게임 스펙: `docs/game-design/*`
- 기술 계약 스펙: `docs/technical/*`
- 현재 구현 상태: `docs/technical/current-implementation.md`
- 협업/작업 규칙: `docs/team/*`, `docs/guides/*`

## 8. 도구별 공통 해석 원칙

- 특정 AI 도구의 내부 기능보다 **저장소 문서와 스펙을 우선**한다.
- 도구가 AGENTS.md를 직접 읽지 못하더라도, 작업자는 이 문서를 기준으로 행동해야 한다.
- 도구별 프롬프트 차이가 있더라도 결과물 형식과 스펙 점검 절차는 통일한다.
